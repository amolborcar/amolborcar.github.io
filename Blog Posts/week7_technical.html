SQL Injection and Why It's a Problem

May 16, 2014

SQL Injection is one of numerous attack techniques that hackers can use to either gain access to, alter, or delete personal information from a database. SQL Injection, put simply, is the act of adding a little extra into an input to a message you're sending to a website in order to cause trouble. When I first read about it, it occurred to me not only how simple it is but also how destructive it can be. I like a lot of people have heard stories of large companies being hacked and people's credit card information or social security numbers being stolen, but I always assumed that this all took place through complex hacks with groups of malicious people working in some shady underground bunker (I watch too many movies). Turns out, that's not necessarily the case. Companies even as large as Microsoft and Yahoo! have fallen prey to what turns out is a pretty simple technique.

SQL Injection works by supplying an input that alters the SQL statements that pull information to give the website. A common example of SQL being used in a website is to process login information (i.e. a username/password). The user would supply his or her username/password, and if those credentials are present in the database, the user would be allowed to continue. A common example I've found online is as follows:
A user inputs a username, which is processed by the following SQL statement:

SELECT * FROM users WHERE name = ' + userName + ';

This code would allow only a userName that is present in the users table to pass through the barrier. However, a person trying to get in without authorized entry could use SQL Injection like this:

SELECT * FROM users WHERE name = ' ' or '1' = '1';

Doing this is remarkably simple. The person would simply enter the following as the username:

' or '1' = '1' --

This would add in '1' = '1' to the WHERE condition in the SQL statement Because '1' = '1' is always True, the condition would evaluation to True every time.

This obviously is a very simple example, and almost every large website (I hope) is now prepared to deal with a simple attack. However, the Open Web Application Security Project (OWASP) listed SQL Injection as one of its top 10 flaws because SQL Injection is such a multifaceted problem. A technique called Blind SQL Injection can be used to detect further vulnerabilities basically by prodding at how the website interacts with its database, and then set the stage for more malicious attacks. Imagine what a disaster it would be if a large company like Amazon lost all of its customer records! Unfortunately, some companies have had to deal with that reality.

Web developers now employ tactics to prevent SQL Injection in their code. One such technique is to only allow input that is of a certain variable type, and fail to recognize another variable type, which most malicious attacks would be. Another similar technique is to disallow any input from the user that contains values that are what you would find in a SQL statement. For example, disallowing single quotes (') which are used in SQL a lot, or not allowing words such as DROP or ALTER. If the code detects any of those, it returns an error. There are now security companies that will audit your website and tell you where it is vulnerable (for a small fee of course), though I think experienced coders should learn to recognize these things for themselves. Of course, this is easier said than done, since no one designs their site to be hackable, but as I learn more and start programming for real I hope to understand all the best ways to prevent SQL Injection on my websites.
