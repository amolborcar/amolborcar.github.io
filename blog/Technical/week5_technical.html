Classes in Ruby

May 4, 2014

As the great Elaine Benes once said, our goal should be a society without classes. Unfortunately, even Ruby isn't perfect in this regard. Every object in Ruby is an instance of some sort of class, whether user-created or built-in. The purpose of a class is to grant objects that have similar qualities access to a common group of methods. An example of a built-in class is the Object class. Every new object that is created has, by default, access to the methods that are made by the Object class. For example, if we create a new object, we can see all methods that can be performed on it:

my_obj = Object.new # This creates a new instance of the Object class.
puts my_obj.methods.sort # This lists out all methods available on the Object class.

Classes are a great way to give properties to new objects based on their characteristics in common with other objects. For example, let's say I've obtained a time machine. With this time machine, I've decided to travel back to 1998 and open a video rental store to get rich. I want to create a record of all movies that my store has in inventory. For this, I'll create a new class called Movie.

class Movie
def initialize(title, year) # create an instance of the class with two inputs.
@title = title
@year = year # Create two instance variables, which in this case are unique to each movie
end
def describe
puts "#{@title} was made in {@year}" end
end
end

movie1 = Movie.new("The Shawshank Redemption", 1994)
movie2 = Movie.new("The Usual Suspects", 1995)

Each movie that I created is a new instance of the class. Initialize is a method that is run automatically when a new object is created. In my case, the initialize method takes the two input variables and creates instance variables, unique to that instance of the class, with their values. The describe method puts the resulting sentence.

So now I have a record of what movies I have in stock. When I have them in the storeroom, I sort them by title and year (hence those variables as inputs). However, when I put them on display for rental, I sort them by genre. Instead of creating a new class that takes the two original variables (title and year) along with a new one (genre), I can create a subclass of Movie that automatically inherits Movie's variables and requires a unique input of its own.

class RentalMovie < Movie
def initialize(title, year, genre) # create an instance of the class with three inputs.
super(title, year) # Inherit the title and year variables from Movie.
@genre = genre
end
def describe
super # this inherits the super method from the Movie class.
end
end

The super command in the above methods is instructing each method to inherit the same method's commands from the parent class. In this case, the parent class of RentalMovie is Movie. So in this case, the program will search through the Movie class for methods named initialize and describe and pass those instructions to the RentalMovie class. So in this case, the initialize method is now creating instance variables for @title and @year, in addition to @genre (which is created directly in the RentalMovie class). The describe method is doing the same thing in RentalMovie that it did in Movie.

This concept is known as inheritance. Inheritance is a good way to make your code DRY (don't repeat yourself) and to help categorizing objects. Organizing objects into classes and subclasses is a good way to make sure that similar methods can be performed on similar objects. I used a movie store example here, but any number of programs could benefit from classes and subclasses. For example, a Zoo might want to track all the animals it houses as the same class, and then be able to perform slightly different methods for different types (or subclasses) of animals (such as mammals vs. amphibians vs. birds). Inheritance is a way to avoid having to create brand new methods for each subclass.
