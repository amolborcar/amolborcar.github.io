What is Rails?

May 25, 2014

Introduction

Dev Bootcamp is focused mainly on teaching web development via Ruby on Rails. This week the technical blog prompt was to research one of a given set of topics. One of the topics was "what is Rails?" I read that and thought "hey, yeah, I'm about to dive into Rails and I really don't even know what it is!" So far we've covered HTML/CSS, JavaScript, Ruby, and SQL, but no Rails, so I did some research and will attempt to explain at a high level what Rails actually is.

Ok, so What is Rails?

Ruby on Rails, or Rails for short, is a web application framework that utilizes the Ruby programming language to build applications. Rails interacts with the bare bones of a web page (HTML, CSS, and sometimes JavaScript) to create a more dynamic web application. Rails runs on a web server, so it is a "back end" platform that performs its functionality behind the scenes (as opposed to on the screen of a web browser such as Chrome or Firefox). I won't go into all the details, but basically languages like Rails (and there are many other comparable frameworks) enable programmers to build much more complicated and deeper websites and applications. Twitter, for example, was originally built on Rails, as were popular sites like Hulu, Pitchfork, and Github.

What Makes Rails Unique?

Rails was released in late 2005 with the goal of making collaboration and maintenance easier than in other frameworks. Ruby was created about a decade earlier, but many of the goals were the same: creating a simple, intuitive, clean programming language that eliminated unnecessary code. Rails' creator, David Heinemeier Hansson, liked this about Ruby and wanted to create a similar framework.

Rails, unlike other web frameworks, has a set of conventions that almost all programmers adhere to. This reduces clutter and unnecessary code found in projects built using other languages. For example, if you create an object in Rails called "User" it will automatically save data to a database table called "users" instead of having the programmer assign a table for the data to be saved to. This assumption means that if a programmer sticks to this convention, he or she can eliminate chunks of code. However, if someone deviates from this convention, then the code either will not work or will become unnecessarily lengthy. I'll admit that at first this seemed counter-intuitive to me: much of what I hear nowadays is about NOT doing things according to convention or "the way they should be done," but a programming language of all things has a set way of doing things. However, when I read the reasoning behind it, it made a lot of sense.

Rails also embraces Ruby's DRY (Don't Repeat Yourself) mentality which makes it easier to collaborate on code. The fact that everyone uses the same conventions means that one person's code will likely be easy to interpret by another Rails programmer. So if a new programmer takes over a particular application, he or she should be able to pick up the code pretty easily.

Why is Rails Gaining in Popularity?

All of the reasons above are reasons why Rails is becoming more popular, but also the growth in popularity of Ruby (again, for reasons described above) lends itself to Rails' growth. Another big advantage is that Rails is open source; this means that programmers can design their own Rails improvements and share them. In addition, one very attractive feature of Ruby is Gems, which are packages for the Ruby language that contain custom programs, modules, or anything really that enable programmers to share code. Ruby Gems increase the scope and abilities of the language, and the network of programmers that use Ruby are always sharing new Gems they've made. As Ruby becomes a more powerful language, it follows that Rails does as well!
